## Что добавлено

1. **Роль пользователя в БД**
   - В таблицу `users` добавлено поле `role` со значением по умолчанию `user`.
   - Роли: `user`, `admin`, `superadmin`.

2. **JWT payload с ролью**
   - Теперь access/refresh токены содержат `role`, чтобы фронт мог быстро понять, кто перед ним.

3. **RBAC middleware**
   - `requireRole([UserRole.Admin])` проверяет роль по базе и блокирует доступ обычным пользователям.
   - `requireRole([UserRole.Superadmin])` используется для управления ролями.

4. **Админ‑эндпоинты**
   - `GET /api/admin/todos` — просмотр всех задач всех пользователей.
   - `GET /api/admin/users` — список пользователей и их ролей.
   - `PUT /api/admin/users/:id/role` — смена роли (только superadmin).
   - `POST /api/admin/boot-superadmin` — bootstrap супер‑админа по секрету.

5. **Frontend**
   - Скрываем ссылку на `/admin` для обычных пользователей.
   - В шапке отображаем имя/почту и аватар.
   - Админ‑страница со списком задач и пользователями (с ролями).

## Как это работает

### Роль в БД
Роль хранится в таблице `users.role`. Это единый источник истины.

### JWT payload
При логине/регистрации/refresh мы кладём роль в токен. Это удобно для фронта и простых проверок UI.

### Проверка прав
Для защищённых эндпоинтов используется `requireRole`:

1. Берём `userId` из `res.locals.auth`.
2. Достаём роль пользователя из БД.
3. Если роль не подходит — возвращаем `403 Forbidden`.

Такой подход защищает от устаревших токенов и подмены роли на клиенте.

### Привязка к коду
- `backend/middleware/requireRole.ts` — центральная проверка роли по базе. Если пользователь не найден — `401`, если роль не подходит — `403`.
- `backend/routes/authRoutes.ts` — формирует JWT с ролью, отдаёт профиль вместе с ролью.
- `backend/routes/adminRoutes.ts` — содержит:
  - `boot-superadmin` (секретный endpoint)
  - `/users` и `/users/:id/role` (управление ролями)
  - `/todos` (просмотр всех задач)
- `backend/types.ts` и `frontend/src/types/admin.ts` — единое объявление `enum UserRole`.

### Почему в SQL используется `todos.` / `users.`
В админ‑эндпоинте используется `JOIN` таблиц `todos` и `users`, обе имеют поле `id`.
Без префикса `todos.id`/`users.id` запрос становится двусмысленным.
Поэтому мы явно указываем таблицу, чтобы избежать ошибок.

## Почему так сделано

- **Безопасность:** роль всегда проверяется по базе.
- **Управляемость:** роль можно менять сервером и не бояться старых токенов.
- **Масштабируемость:** легко добавить новые роли/права.

## Как использовать

### 1. Назначение superadmin (bootstrap)
Используется закрытый эндпоинт и секрет:

```
SUPERADMIN_BOOTSTRAP_SECRET=your_secret_here
```

Пример запроса:
```
curl -X POST http://localhost:3001/api/admin/boot-superadmin \
  -H "Content-Type: application/json" \
  -H "x-bootstrap-secret: your_secret_here" \
  -d '{"email":"user@example.com"}'
```

Эндпоинт нужен только один раз, чтобы назначить первого superadmin.

### 2. Админ‑доступ
Admin и superadmin могут открыть:
- `/admin` на фронте
- `GET /api/admin/todos` в API

Обычный пользователь получит `403 Forbidden`.

### 3. Управление ролями
Только superadmin может менять роли:

```
PUT /api/admin/users/:id/role
{ "role": "admin" }
```

### Защита от самодемота
Если superadmin пытается снять роль с самого себя, бэкенд возвращает ошибку.
Это сделано, чтобы в системе всегда оставался хотя бы один superadmin.
