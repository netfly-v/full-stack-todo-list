## Подробное объяснение изменений (backend)

Ниже — детальный разбор того, что именно изменилось, зачем это нужно и как это работает.  
Фокус на файлах, которые я менял: `backend/migrate.ts`, `backend/migrations/*.sql`, `backend/server.ts`, `backend/routes/authRoutes.ts`, `backend/routes/todoRoutes.ts`, `backend/types.ts`, `backend/db.ts`.

---

## 1) `backend/migrate.ts` — минимальный раннер миграций

Задача: заменить «ручное создание таблиц при старте» на управляемые миграции.

### Что такое миграции
Миграция — это SQL‑файл, который описывает один шаг изменения схемы.  
Например: «создать таблицу», «добавить колонку», «добавить индекс».  
Так схема базы хранится в виде истории изменений, а не в виде одного «большого» скрипта.

### Что делает `migrate.ts`
Файл запускается как обычный скрипт (`npm run migrate`) и делает:

1. **Создает таблицу `schema_migrations`**, где хранятся имена уже примененных миграций.
2. **Считывает все файлы** из `backend/migrations`.
3. **Сортирует их по имени** (поэтому важна нумерация `001_`, `002_` и т.д.).
4. **Применяет только те миграции**, которых нет в `schema_migrations`.
5. **Каждую миграцию запускает в транзакции**, чтобы при ошибке откатить частичные изменения.

Ключевые участки:

- `ensureMigrationsTable()` — гарантирует таблицу для истории миграций.
- `getAppliedMigrations()` — читает уже выполненные миграции.
- `runSingleMigration()` — оборачивает SQL в `BEGIN/COMMIT`, при ошибке делает `ROLLBACK`.
- `runMigrations()` — запускает цикл по файлам.
- Внизу файла есть «CLI‑режим»:
  - если файл запущен напрямую, применяем миграции и выходим.

Таким образом, схема больше не создается «магически» на старте через `initDb()`,
а управляется через `migrations/*.sql`.

## 2) `backend/migrations/001_init.sql` — начальная схема

Первый файл миграций — база, от которой мы отталкиваемся:

- Таблица `users`:
  - `id` — идентификатор (SERIAL).
  - `email` — уникальный логин.
  - `password_hash` — хэш пароля.
  - `created_at` — время создания.
- Таблица `todos`:
  - `id`, `title`, `description`, `completed`, `created_at`.

Это эквивалент старому `initDb()`, только теперь хранится в виде версии схемы.

## 3) `backend/migrations/002_add_todos_user_id.sql` — связь todos с users

Эта миграция делает todos «привязанными» к пользователю:

1. Добавляет колонку `user_id` (обязательная, NOT NULL).
2. Вешает внешний ключ на `users(id)` с `ON DELETE CASCADE`.
3. Создает индекс по `user_id`, чтобы запросы «все тудушки пользователя» были быстрыми.

`ON DELETE CASCADE` означает: если удалить пользователя, его задачи удалятся автоматически.

Важно: так как `user_id` NOT NULL, в базе не должно быть старых todo без владельца.
Поэтому удобнее пересоздать БД при первом переходе.

## 4) `backend/server.ts` — запуск миграций при старте

Раньше было:
- `initDb()` — создавал таблицы при запуске.

Теперь:
- `runMigrations()` — применяет все новые миграции.

Это делает запуск стабильным:
- если миграции уже были, ничего лишнего не выполняется;
- если добавлена новая миграция — она применится автоматически.

---

## 5) `backend/db.ts` — убран `initDb()`

В `db.ts` оставлен только пул подключений к Postgres.
Создание таблиц вынесено в миграции.

Это упрощает ответственность файла:
- `db.ts` = подключение к БД;
- миграции = схема.

---

## 6) `backend/routes/authRoutes.ts` — экспорт `requireAuth`

Функция `requireAuth` раньше была локальной.
Теперь она экспортируется, чтобы использовать ее в других роутерах:

- проверяет cookie `access_token`;
- валидирует JWT;
- кладет payload в `res.locals.auth`;
- если невалидно — возвращает 401.

---

## 7) `backend/routes/todoRoutes.ts` — «user‑scoped» todos

Главная идея: все операции над тудушками должны фильтроваться по `user_id`.
Для этого:

1. Весь роутер теперь защищен `router.use(requireAuth);`
2. В каждом обработчике мы читаем `auth.userId` из `res.locals.auth`.
3. SQL‑запросы добавляют условие `user_id = $X`.

Примеры:

- **GET /api/todos**:
  - было: `SELECT ... FROM todos`
  - стало: `SELECT ... FROM todos WHERE user_id = $1`
  - фильтр `active/completed` добавляется через `AND`.

- **GET /api/todos/:id**:
  - добавлено `AND user_id = $2`.
  - теперь нельзя получить чужую задачу.

- **POST /api/todos**:
  - при вставке добавляем `user_id`.
  - задача сразу принадлежит текущему пользователю.

- **PUT /api/todos/:id** и **DELETE**:
  - в `WHERE` добавлено `AND user_id = $2`.
  - нельзя обновить/удалить чужую задачу.

Также в ответы добавлено поле `created_at as "createdAt"` — это удобнее в JS.

## Итоговая логика потока

1. Пользователь логинится → получает cookies с access token.
2. Каждый запрос к `/api/todos` проходит через `requireAuth`.
3. `requireAuth` кладет `userId` в `res.locals`.
4. SQL‑запросы используют этот `userId`.
5. В итоге каждый пользователь видит только свои задачи.
